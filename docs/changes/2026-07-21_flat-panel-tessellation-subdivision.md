# CAD import 시 평평한 큰 면을 잘게 재분할 (ROI 박스 드래그가 항상 컴포넌트 전체를 잡던 문제)

> 2026-07-22 통합 업데이트: 아래의 50/10 mm² 수치는 초기 실험 기록이며
> 현재 설정이 아니다. 통합본은 CAD 대각선 기준 목표 edge(대각선 약 512분할,
> 0.5–3.0 mm 범위)와 최대 750,000 face 예산을 함께 사용하는 적응형 정책으로
> 교체했다. 따라서 모든 CAD에 동일한 10 mm² 기준을 적용하지 않는다.

## 배경
ROI 박스 드래그로 여러 위치를 드래그해도 면적이 항상 같은 숫자(예: 364000)로 나오고, 좁게 드래그해도 ROI View에 큰 삼각형 하나가 그대로 보이는 문제가 계속 재현됐다. 앞서 두 차례(AABB 겹침 → centroid 포함 → 실제 기하학적 삼각형-사각형 교차)로 판정 로직 자체를 계속 고쳤지만 증상이 안 없어져서, 사용자가 직접 확인한 결과 **"COMPONENT의 전체 면적으로 잡히고 있다"**는 걸 확인했다.

**진짜 원인은 판정 로직이 아니라 CAD tessellation(삼각화) 단계였다.** `BRepMesh_IncrementalMesh`의 deflection(허용 오차) 파라미터는 **곡률 오차**만 제어한다 - 완전히 평평한 면(디퓨저 시트, LGP 같은 평판)은 곡률이 0이라서, deflection을 아무리 작게 잡아도 OCC가 그 면을 최소 삼각형 개수(보통 사각형 하나당 삼각형 2개)로만 삼각화하고 더 잘게 안 쪼갠다. ROI는 "Face 단위로만 클립, 삼각형 내부 재분할은 안 함"이라는 설계라서, 삼각형 1~2개가 곧 그 컴포넌트 전체 표면이면 "Face 단위 클립"이 사실상 "컴포넌트 전체 포함"이 되어버렸다.

사용자에게 실제 다각형 클리핑(정밀하지만 구현 복잡, ray trace 백엔드까지 반영 필요) vs CAD import 시 테셀레이션을 더 잘게 재분할(기존 ROI 판정 로직은 그대로, import 단계만 보완) 두 옵션을 제시했고, **테셀레이션 재분할**을 선택함.

## 변경 사항

### `src/leakage_simulator/geometry.py`
- `subdivide_flat_mesh(mesh, max_area_mm2, max_depth=6)` 추가: 면적이 `max_area_mm2`를 넘는 face를 모서리 중점을 이어 4개로 재귀 분할(quadrisection)한다 - 평면 삼각형의 모서리 중점은 항상 같은 평면 위에 있으므로, 이 과정은 원래 형상을 완전히 보존하고 테셀레이션 밀도만 바꾼다.
  - 결과로 나온 모든 하위 face는 원래 face의 `material_id`/`metadata`(component id 등)를 그대로 물려받는다 - 여러 하위 face가 같은 컴포넌트 메타데이터를 공유하는 건 정상이며, `components.py`의 `build_face_groups()`가 원래도 컴포넌트당 여러 face를 다루도록 되어 있어서 그대로 잘 맞는다.
  - 정점은 좌표 기준으로 중복 제거(dedup)해서, 서로 맞닿은 두 큰 삼각형을 각각 재분할해도 공유 모서리의 중점이 같은 정점으로 합쳐진다 (갈라진 틈 없음, 정점 수 낭비 없음).
  - `max_depth`로 최악의 경우(면적 임계값이 아주 작을 때) 재귀가 무한정 깊어지는 걸 방지.

### `src/leakage_simulator/importers.py`
- `ROI_SUBDIVISION_MAX_AREA_MM2 = 50.0` 상수 추가 (약 7mm x 7mm 칸 크기 - ROI 드래그로 고를 만한 최소 단위 감안).
- `_import_step_ocp()`(실제 사용 중인 경로)와 `_import_step()`의 cadquery 경로(현재 환경에선 cadquery 자체가 막혀있어 거의 안 쓰이지만 일관성을 위해) 양쪽 다, 최종 mesh를 반환하기 직전에 `subdivide_flat_mesh()`를 거치도록 추가.
- `_guess_receiver_faces()` 호출은 **재분할 이후**의 mesh를 기준으로 다시 수행 - 재분할 전 face 개수 기준으로 만들어졌던 `receiver_faces`(모든 face 중 13번째마다 하나씩 뽑는 휴리스틱) 리스트는 재분할로 face 개수가 완전히 달라지므로 그대로 쓰면 안 맞는 face를 가리킬 수 있었음 - `_guess_receiver_faces`가 항상 비어있지 않은 리스트를 반환하므로 실질적으로 항상 이 새 리스트로 대체됨.

## 자동 검증
- `tests/test_geometry_subdivision.py` (신규, 5개, 전부 통과): 작은 face는 그대로 유지, 큰 face는 임계값 이하로 쪼개짐, 쪼개진 뒤에도 material/metadata 유지, **전체 면적이 보존됨**(쪼개기 전후 합산 면적 동일), 서로 맞닿은 두 큰 삼각형의 공유 모서리 중점이 중복 없이 하나로 합쳐짐, `max_depth`가 재귀 깊이를 제한함.
- 기존 `tests/test_raytrace_bridge.py`(8개)/`tests/test_roi_native_selection.py`(10개) 회귀 없이 전부 통과 (총 23개).
- **실제 STEP 파일로 end-to-end 확인** (`samples/tv_leakage_full_assembled_no_gap.stp`): 재분할 전 116 face → 재분할 후 **48,602 face**, 최대 face 면적 46.875 mm²(임계값 50 이하로 정상 반영), import+재분할 소요 시간 0.21초. `/api/scene` 엔드포인트를 통해 실제로 이 48,602-face mesh가 그대로 내려오는 것도 확인함 (component_count=4, synthetic=False).

## 후속 조정: 기준을 50 mm² → 10 mm²로 낮춤
브라우저 실사용 확인 결과, ROI View에서 박스 경계에 삐죽삐죽 튀어나오는 부분이 눈에 띄었음 - 이건 버그가 아니라 "삼각형 단위로 통째 포함/제외"(폴리곤 클리핑 없음) 설계상 남는 근사치 오차이고, 재분할 칸 크기(그 시점 50mm² → 약 7mm)만큼 튀어나오는 게 정상 동작임. 오차를 줄이려고 기준을 `10.0`으로 낮춤(칸 크기 약 3~4mm).

실제 STEP 파일로 재확인한 face 개수/처리 시간:

| 기준 (mm²) | face 개수 | import 시간 |
|---|---|---|
| 50.0 | 48,602 | 0.23s |
| 20.0 | 75,104 | 0.26s |
| 10.0 (현재 적용) | 112,640 | 0.43s |
| 5.0 | 202,112 | 0.73s |

import 속도는 10mm² 기준에서도 충분히 빠름 - 브라우저 렌더링/드래그 판정 체감 속도는 실사용으로 확인 필요.

## 현재 한계
- **브라우저 실사용 테스트 아직 없음**: 이번에도 Python 스크립트 + `/api/scene` curl 테스트까지만 확인했고, 실제로 브라우저에서 여러 위치에 드래그했을 때 면적이 위치마다 다르게 나오는지, 10mm² 기준에서 드래그 반응 속도가 체감상 괜찮은지는 사용자가 직접 확인해야 함.
- `max_area_mm2 = 10.0`도 여전히 임의로 고른 값 - 더 정밀해야 하면 낮추고(단 face 개수/속도 트레이드오프 있음), 반대로 느리면 다시 높일 수 있음.
- 완전한 다각형 클리핑(면 일부만 정확히 잘라내기)은 여전히 안 함 - 지금은 "재분할된 작은 삼각형 단위로 통째 포함/제외"이므로, 재분할 격자 크기(현재 약 3~4mm)만큼의 경계 오차는 설계상 항상 남음. 이 오차가 실용적으로 충분히 작은지는 실사용 확인 필요.
- Gap/Emitter/Receiver 등 다른 기능이 face index(예: `reference_vertex_indices`, Gap의 `target_face_indices`)를 직접 참조하고 있었다면, 재분할로 face 개수/인덱스가 완전히 달라지므로 **새로 CAD를 로드한 뒤부터는 영향받지만, 이미 저장된 기존 세션/설정에 남아있던 face index 참조는 새 인덱싱과 안 맞을 수 있음** - 새로 로드한 세션에서는 문제없음.
