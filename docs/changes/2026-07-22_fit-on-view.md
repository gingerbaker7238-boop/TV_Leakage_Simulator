# "Fit on View" 마우스/단축키 기능 추가

## 요청
"마우스 조작 기능 추가하나 하고 싶어. FITONVIEW가 있으면 좋겠어. ROI VIEW에서도 동일하게 적용될 수 있게. 단축키는 어떤게 좋을지 제안해줘"

기존 "Fit" 버튼은 `LeakageThreeViewer.fit()`을 호출하는데, 이건 시점(각도)까지 고정 iso 뷰로 리셋하면서 크기만 모델에 맞춤 - 사용자 확인 결과 원하는 건 **지금 보고 있는 각도는 그대로 두고, 줌/팬만 조정해서 화면에 꽉 차게 하는 것**(일반 CAD 툴의 "Zoom to Fit"과 동일한 동작).

## 변경 사항
- `LeakageThreeViewer.fitToViewKeepingOrientation()` 추가: 현재 카메라 방향(타겟→카메라 벡터)은 유지한 채, 실제로 렌더링된 지오메트리의 `boundingSphere`를 기준으로 카메라를 그 방향을 따라 밀거나 당겨서(FOV 기준 구를 꽉 채우는 거리) 재배치. `controls.target`도 그 중심으로 옮김.
  - **ROI View에서도 동일하게 동작**: bounding sphere를 `surface.geometry`(이미 `buildBufferGeometry(meshRef, roiFaces, {{excludeFaces: hiddenFaces}})`로 ROI/Hide 필터가 반영되어 만들어진 지오메트리)에서 직접 읽으므로, ROI View가 지금 실제로 보여주고 있는(잘린) 영역 기준으로 정확히 맞춰짐 - 기존 "Fit" 버튼이 쓰는 `bboxCenterAndSize(meshRef)`는 필터링 안 된 원본 mesh 전체 기준이라 이 부분에서 서로 다름.
- **단축키: `F` 키** - Fit(맞춤)의 이니셜이라 외우기 쉽고, 이 앱에 아직 전역 단축키가 하나도 없어서 겹칠 것도 없음. 텍스트 입력 칸(ROI 이름 등)에 포커스가 가 있을 때는 무시하도록 처리 - 이름에 "f"를 입력하다가 뷰가 갑자기 바뀌는 일 없음.
- **마우스: 뷰어 더블클릭**으로도 동일하게 트리거 (Full View/ROI View 각각 독립적으로, 더블클릭한 쪽만). 다른 CAD/3D 뷰어에서 흔히 쓰는 제스처.
- `F` 키는 Full View + ROI View 둘 다 한 번에 맞춤 (기존 카메라 프리셋 버튼들이 항상 양쪽에 동시 적용되는 것과 동일한 패턴).
- 뷰어 하단 안내 문구("Drag = rotate, ...")에 "F/Double-click = fit view." 추가.

## 검증
- Python 문법 검사, 서버 재부팅(HTTP 200), 새 코드가 서빙되는 것 확인.

## 버그 수정: 화면이 가득 안 차던 문제
1차 실사용 확인: "FITONVIEW를 해도 화면이 가득차지 않고, 잘 보이지 않아".
- **원인**: 처음 구현은 bounding **sphere**(구) 기준으로 카메라 거리를 계산했음. 구는 bounding box의 대각선 전체를 감싸야 하는데, 얇고 넓적한 형태(TV 섀시처럼 폭/높이에 비해 두께가 얇은 조립체)는 대각선이 실제 폭/높이보다 훨씬 커서, 구를 화면에 맞추면 실제 내용물(사각형)은 그보다 훨씬 작게 보이고 주변에 여백이 많이 남음.
- **수정**: bounding **box**를 카메라의 현재 right/up 축에 실제로 투영한 폭/높이를 계산해서(현재 보는 각도 기준으로 정확한 가로/세로 크기), 그 값이 각각 수평/수직 FOV에 딱 맞도록 거리를 계산 - 더 타이트하게, 실제로 화면을 채움. 여백은 8%로 줄임(전엔 15%).

## 함께 반영: ROI 추가 중/완료 후 큰 화면 자동 전환
"ROI 설정 후에는 큰 화면이 ROI VIEW가 되고, FULL VIEW가 작은 화면으로... ROI 추가 설정 시에는 큰화면이 FULL VIEW가 되고 이때는 작은 화면은 필요없어" 요청 반영.
- 예전에 `updateViewerMode()`가 "ROI 선택이 하나라도 있으면 무조건 ROI View를 큰 화면으로 승격"했다가, 첫 ROI 추가 직후 큰 화면이 그 ROI로 잘린 영역이 되어버려서 두 번째 ROI를 전체 모델 아무 데나 드래그로 못 만드는 버그가 있었음 (그래서 한 번 "Full View 항상 큰 화면"으로 고정했었음).
- 이번엔 승격 조건을 **"ROI 드래그 무장(armed) 중이 아닐 때"**로 제한: `+ ROI 추가`를 눌러 무장하는 순간(armed=true) 무조건 Full View가 큰 화면이 되고(드래그로 모델 전체에 접근 가능), 드래그가 끝나 자동으로 무장 해제되면 그 시점에 활성 ROI가 있으면 ROI View가 큰 화면으로 승격됨. 두 요구사항이 충돌하지 않고 자연스럽게 전환됨.
- `updateRoiBoxDragArmUI()`(무장 상태가 바뀔 때마다 호출됨)와 `resetRoiSelection()`(CAD 재로드 시)에도 `updateViewerMode()` 호출을 추가해서, 무장/해제·리셋 즉시 화면 전환이 반영되도록 함.

## 버그 수정: 두 번째 ROI부터 드래그가 항상 "결과 없음"으로 나오던 문제
2차 실사용 확인: 큰/작은 화면 자동 전환은 마음에 든다고 확인 - 그런데 ROI 하나 만든 뒤 두 번째를 추가하려고 드래그하면 "ROI 선택 결과: 없음" 팝업만 뜸.
- **원인**: "+ ROI 추가"를 누르는 순간 `updateViewerMode()`가 Full View를 큰 화면으로 즉시 전환하는데(위 항목), 이 화면 크기 변경은 CSS가 즉시 반영되지만 각 `LeakageThreeViewer`의 카메라 종횡비(`camera.aspect`)를 갱신하는 `ResizeObserver`는 **비동기로(다음 프레임에) 실행**된다. 사용자가 버튼을 누르자마자 바로 드래그를 시작하면, `computeThreeXyBoxFromScreenRect()`가 이미 커진 DOM 크기와 아직 갱신 안 된(예전 작은 화면 기준) `camera.aspect`를 섞어서 계산하게 되어 완전히 엉뚱한 위치로 언프로젝션됨 - 그래서 실제로는 뭔가 위에 드래그했어도 결과가 항상 빈 값으로 나왔음.
- **수정**: `updateViewerMode()`가 화면 전환(큰/작은 교체) 직후 `threeFullRenderer.resize()`/`threeRoiRenderer.resize()`를 **동기적으로 즉시** 호출하도록 추가 - 비동기 ResizeObserver를 기다리지 않고 카메라 종횡비를 그 자리에서 바로 맞춤.

## 버그 수정 2: 첫 시도(resize() 동기 호출)로는 부족했음 - 진짜 원인은 CSS transition
3차 실사용 확인: 여전히 두 번째 ROI부터 "결과 없음". `resize()`를 동기 호출로 바꿨는데도 재현됨.
- **진짜 원인**: `.viewer-card`에 `transition: all 220ms ease`가 걸려 있어서, 큰/작은 화면 교체가 순간적으로 바뀌는 게 아니라 **220ms에 걸쳐 슬라이드 애니메이션**된다. `viewerStage.className`을 바꾼 "직후"에 동기적으로 `resize()`를 불러도, 그 시점의 실제 레이아웃은 애니메이션 시작 지점(옛날 크기에 가까움)이라 여전히 틀린 크기를 읽는다 - 그래서 사용자가 버튼을 누르고 220ms 안에 드래그를 시작하면 여전히 같은 문제가 재현됨.
- **진짜 수정**: 이 프로그램적인 화면 교체 순간에만 `.viewer-card-no-transition`(=`transition: none !important`) 클래스를 잠깐 붙여서 애니메이션 없이 즉시 스냅되게 만들고, 그 상태에서 `resize()`로 정확한 최종 크기를 읽은 뒤, 다음 프레임에 그 클래스를 다시 떼어내 원래의 부드러운 전환 애니메이션은 다른 상황(예: 수동으로 Full/ROI 전환 버튼 누를 때 등)에서는 그대로 유지되도록 함.

## 버그 수정 3 (진짜 최종 원인): disarm 호출 순서
4차 실사용 확인 - 여전히 재현됨. 콘솔에 임시 디버그 로그를 추가해서 실제 값을 하나씩 대조한 끝에 정확한 원인을 찾음 (트랜지션/리사이즈 타이밍 문제가 아니었음).

- **원인**: `selectRoiFacesInThreeRect(mode, rect)`(마우스를 뗀 순간 실행되는, 실제 박스 판정 함수)가 맨 첫 줄에서 `disarmRoiBoxDrag()`를 호출하고 있었다. 그런데 `disarmRoiBoxDrag()`는 `updateViewerMode()`를 호출하고, 이미 ROI가 하나라도 있는 상태(=두 번째 ROI를 추가하는 중)에서는 이 호출이 **즉시 Full View를 다시 작은 화면으로 되돌린다** (ROI View 승격). 즉, 정작 드래그한 화면 좌표(`rect`)를 실제 월드 좌표로 변환하는 `computeThreeXyBoxFromScreenRect()`가 실행되는 시점엔 **이미 화면이 작아진 뒤**라서, 사용자가 큰 화면 기준으로 드래그한 픽셀 좌표를 완전히 다른(작은 화면) 기준으로 잘못 변환하고 있었다. 실제로 콘솔 로그에서 두 번째 드래그 직전에 `updateViewerMode(... addingRoi=false ...)`가 한 번 더 찍히면서 화면이 278x199.5(작은 크기)로 줄어드는 게 그대로 확인됨.
- **수정**: `disarmRoiBoxDrag()` 호출을 함수 **맨 끝(`finally` 블록)으로 이동** - 박스/Face 계산을 다 마친 뒤에야 화면이 줄어들도록 순서를 바꿈. 화면이 한 프레임 늦게 줄어드는 건 눈에 안 보이지만, 계산 자체는 항상 올바른(드래그 당시) 화면 크기 기준으로 하게 됨.
- 이 과정에서 추가했던 임시 콘솔 로그(`[ROI DEBUG]`)는 전부 제거함.

## 검증 (추가)
- **브라우저 실사용 테스트 필요**: 회전해둔 상태에서 F 또는 더블클릭 시 화면을 꽉 채우는지, "+ ROI 추가" 누르면 즉시(애니메이션 없이) Full View가 커지고, 드래그 완료 후 ROI View가 커지는지, 그리고 이번엔 정말로 두 번째/세 번째 ROI도 드래그한 위치대로 정상적으로 잡히는지 사용자 확인 대기 중.

## 후속 개선 1: 좌표축 삼각대(axis triad)가 Fit on View 후 너무 커 보이던 문제
전부 해결 확인 후, "fitonview 할때 좌표계도 너무 많이 커지는데 좀 줄여줘" 요청.
- **원인**: 좌표축 표시(`this.axis`) 크기가 `this.size`(모델 전체 bounding box 크기)에만 비례해서 고정된 월드-공간 크기로 계산되고 있었음 - 카메라 줌(거리)과는 무관. Fit on View가 이전보다 훨씬 타이트하게(가깝게) 맞추도록 고쳤기 때문에, 같은 월드-공간 크기의 좌표축이 화면상으로는 상대적으로 훨씬 커 보이게 된 것.
- **수정**: 좌표축 크기를 "현재 카메라 거리 기준으로 화면에서 차지하는 비율이 항상 일정하도록" 다시 계산하도록 변경(`updateAxisScale()`) - 카메라와 모델 중심 사이 거리, FOV로 그 위치에서 실제로 보이는 화면 높이를 구하고, 그 10%로 축 크기를 정함. 매 프레임(`animate()`)마다 갱신되도록 해서 줌/팬/회전 어떤 걸 해도 좌표축이 항상 비슷한 화면 크기를 유지함 (기존의 확대/축소 시 좌표축 슬라이더는 그대로 추가 배율로 작동).

## 후속 개선 2: ROI View 작업 중 Full View 미리보기가 거슬리던 문제
"ROI VIEW를 봐야 할때 FULL VIEW가 작업하는데 거슬리는데, 사이즈 20% 줄여주고, 화면 이동/사이즈 확대 축소도 될 수 있게 반영해줘" 요청.
- ROI View가 큰 화면일 때(`mode-roi`) Full View가 차지하는 작은 코너 패널 크기를 20% 축소 (`31% width/240px height` → `24.8%/192px`).
- 팬(오른쪽 드래그)/줌(휠)은 각 뷰어 인스턴스가 독립적인 `OrbitControls`를 갖고 있어서 패널 크기와 무관하게 이미 동작함 - 별도 코드 변경 없이 확인만 함.

## 후속 개선 3: 작은 코너 패널을 직접 옮기고 크기 조절할 수 있게
"작은 화면 FULL VIEW에서의 마우스 조작 기능 모두 반영되어 작동된다. 내가 요청한 건은 작업 화면 자체를 시뮬레이터 큰 화면에서 자유롭게 이동 및 화면 사이즈 조정이 필요하다" - 3D 카메라 팬/줌이 아니라, **패널(창) 자체**를 옮기고 크기 조절하는 요청이었음.
- 작은 코너 패널(현재 큰 화면이 아닌 쪽 - Full CAD View 또는 ROI View)의 **헤더(제목 부분)를 드래그하면 이동**, **오른쪽 아래 모서리의 그립을 드래그하면 크기 조절**되도록 추가.
- 이동/크기 조절 중에는 (다른 프로그램적 화면 전환과 마찬가지로) `.viewer-card-no-transition`으로 슬라이드 애니메이션을 잠깐 꺼서 커서를 그대로 따라가도록 함.
- 크기 조절 중에는 매 순간 해당 뷰어의 `.resize()`를 호출해서 3D 렌더링이 실시간으로 새 크기에 맞춰짐.
- 마지막으로 옮기고 조절한 위치/크기를 `state.smallPanelPlacement`에 기억해뒀다가, 그 패널이 다시 "작은 코너" 역할이 될 때 그대로 복원함 - 매번 기본 코너 위치로 리셋되지 않음.
- 반대로 그 패널이 "큰 화면" 역할이 될 때는 인라인 스타일을 지워서 원래 CSS(`inset: 0`)가 정상적으로 전체를 채우도록 함.
- 드래그는 화면(`.viewer-stage`) 영역을 벗어나지 않도록 경계 안에서만 움직이도록 제한.
